1. Introduction

JAVL.jl is a pure Julia implementation of Mark Drela's AVL (Athena Vortex Lattice) 3.52 aerodynamic solver. It implements the Vortex Lattice Method (VLM) for computing aerodynamic characteristics of arbitrary wing/body configurations in subsonic, inviscid, incompressible (with Prandtl-Glauert correction) flow.

The solver computes:

  • Total force and moment coefficients (CL, CD, CY, Cl, Cm, Cn)
  • Stability and control derivatives
  • Trefftz-plane induced drag and span efficiency
  • Strip-wise loading distributions (c*cl, local Cl, induced angles)
  • Hinge moments for control surfaces
  • Nodal forces for FEM coupling

JAVL.jl reads the same .avl input files as the original Fortran code, making it a drop-in replacement for AVL's computational core. It has been validated against Fortran AVL 3.52 across 55 test cases, with CL agreement within 1% on 47 of 54 solvable cases (the remaining discrepancies are due to BODY elements not yet fully implemented).

JAVL.jl is released under the MIT License (open source).

Note: JAVL.jl aims for numerical equivalence with Fortran AVL 3.52. The same algorithms are used: horseshoe vortex kernel, cubic spline interpolation, Kutta-Joukowski force integration, and Trefftz-plane drag computation.

2. Installation

JAVL.jl has no external package dependencies beyond the Julia standard library (LinearAlgebra and Printf). Any Julia 1.6+ installation is sufficient.

Obtaining the source

# Clone the repository
git clone https://github.com/your-org/JAVL.jl.git

Loading into Julia

# From the Julia REPL or a script:
push!(LOAD_PATH, "path/to/JAVL.jl/src")
include("path/to/JAVL.jl/src/AVL.jl")
using .AVL

Alternatively, you can activate the project environment if a Project.toml is present: 

# From the Julia REPL:
using Pkg
Pkg.activate("path/to/JAVL.jl")
include("path/to/JAVL.jl/src/AVL.jl")
using .AVL

3. Quick Start

3.1 Using the high-level driver

The simplest way to run JAVL.jl is through the solve_case driver function. It reads an AVL input file, builds the vortex lattice, and solves for a given angle of attack or target CL in a single call. 

include("src/AVL.jl")
using .AVL

# Solve at alpha = 5 degrees
sol = AVL.solve_case("validation/cases/vanilla.avl"; alpha=5.0)

# Or trim to a target CL
sol = AVL.solve_case("validation/cases/vanilla.avl"; cl_target=0.6)

# Access results
println("CL  = ", sol.cl)
println("CD  = ", sol.cd)
println("Cm  = ", sol.cmy)
println("CDi = ", sol.cdi)
println("e   = ", sol.spanef)

3.2 Step-by-step workflow

For more control over the solver pipeline (e.g., solving multiple run cases with the same AIC factorization, or extracting intermediate data), use the individual functions: 

include("src/AVL.jl")
using .AVL

# 1. Load configuration from .avl file
config = AVL.read_avl("validation/cases/vanilla.avl")

# 2. Build vortex lattice
vl = AVL.build_lattice(config)
println("Surfaces: $(vl.nsurf), Strips: $(vl.nstrip), Vortices: $(vl.nvor)")

# 3. (Optional) Set up body source model if bodies present
src_u_kw = nothing
wcsrd_u  = nothing
if !isempty(config.bodies)
    _src_u, _dbl_u, _wcsrd_u, _wvsrd_u = AVL.setup_body!(vl, config)
    src_u_kw = size(_src_u, 1) > 0 ? _src_u : nothing
    wcsrd_u  = size(_wcsrd_u, 2) > 0 ? _wcsrd_u : nothing
end

# 4. Build and LU-factorize AIC matrix
aic = AVL.setup_aic(vl, config)

# 5. Solve for unit-freestream circulations
gam_u0, gam_u_d = AVL.solve_unit_rhs(vl, config, aic; wcsrd_u=wcsrd_u)

# 6. Set up operating point
sol = AVL.AVLSolution(vl.nvor, vl.ncontrol, vl.nstrip)
rc  = AVL.RunCase(vl.ncontrol)
rc.icon[AVL.IVALFA]   = AVL.ICALFA
rc.conval[AVL.ICALFA] = deg2rad(5.0)  # alpha = 5 degrees

# 7. Execute case (Newton iteration for trim)
AVL.exec_case!(sol, vl, config, aic, gam_u0, gam_u_d, rc;
               src_u=src_u_kw)

# 8. Print results
println("CL  = ", sol.cl)
println("CD  = ", sol.cd)
println("Cm  = ", sol.cmy)
println("CDi = ", sol.cdi)
println("e   = ", sol.spanef)

3.3 Extracting nodal forces for FEM coupling

# After solving...
forces = AVL.get_nodal_forces(sol, vl, config; qinf=0.5*1.225*50.0^2)

# Each element is (x, y, z, Fx, Fy, Fz)
for (x, y, z, fx, fy, fz) in forces
    @printf("  (%.4f, %.4f, %.4f)  F=(%.4f, %.4f, %.4f) N\n",
            x, y, z, fx, fy, fz)
end

4. Input File Format (.avl)

JAVL.jl reads the same .avl geometry files as Fortran AVL 3.52. Lines beginning with ! or # are treated as comments. The file is read top-to-bottom; keywords trigger parsing of subsequent data lines.

4.1 Header

The first non-comment lines define the global configuration:

Plane Vanilla                ! Title string (free-form)
0.0                          ! Mach number
0     0     0.0              ! iYsym  iZsym  Zsym
9.0   0.9   10.0             ! Sref   Cref   Bref
0.50  0.0   0.0              ! Xref   Yref   Zref
0.010                        ! CDo  (optional baseline profile drag)
FieldDescription
MachFreestream Mach number (Prandtl-Glauert correction applied when > 0)
iYsymY-symmetry flag: 1 = symmetric, -1 = antisymmetric, 0 = none
iZsymZ-symmetry flag: 1 = symmetric, -1 = antisymmetric, 0 = none
ZsymZ-coordinate of the ground plane (used with iZsym)
SrefReference area for force coefficient normalization
CrefReference chord for moment coefficient normalization
BrefReference span for moment coefficient normalization
Xref, Yref, ZrefMoment reference point (center of gravity)
CDoBaseline profile drag coefficient (optional, defaults to 0)

4.2 SURFACE definition

Each lifting surface begins with the SURFACE keyword:

SURFACE
Wing                         ! Surface name
8   1.0   12   1.0           ! Nchord  Cspace  [Nspan  Sspace]
ParameterDescription
NchordNumber of chordwise horseshoe vortex panels
CspaceChordwise spacing parameter (1.0 = cosine, 0.0 = uniform, -1.0 = sine, etc.)
NspanNumber of spanwise strips (optional; overrides section-level values)
SspaceSpanwise spacing parameter (optional)

Surface keywords (placed between SURFACE header and first SECTION)

YDUPLICATE
Followed by one value: Ydupl. Mirrors the entire surface about Y = Ydupl. This creates a second surface with all Y-coordinates reflected. Typically 0.0 for left/right symmetry.
SCALE
Followed by three values: Xscale Yscale Zscale. Scales all section coordinates by these factors before lattice construction.
TRANSLATE
Followed by three values: dX dY dZ. Translates all section leading-edge positions by these offsets.
ANGLE
Followed by one value: Ainc (degrees). Adds a global incidence angle to all sections in this surface.
NOWAKE
No data line. Suppresses wake shedding from this surface. Useful for modeling fins or struts that should not generate a wake.
NOALBE
No data line. Removes the freestream alpha/beta influence on this surface. The surface is only affected by rotation rates and control deflections.
NOLOAD
No data line. Excludes this surface from total force/moment summation. Forces are still computed but not added to the global totals.
COMPONENT
Followed by one integer: Ncomp. Assigns this surface to component group Ncomp. Surfaces in the same component use a tighter vortex core radius among themselves.

4.3 SECTION definition

Each section within a surface defines a spanwise station. At least two sections are required per surface to define the spanwise extent. 

SECTION
!Xle    Yle    Zle     Chord   Ainc  Nspan  Sspace
0.0     0.0    0.0     1.0     2.0   0      0
ParameterDescription
Xle, Yle, ZleLeading-edge position of this section
ChordChord length at this section
AincLocal incidence angle (degrees), added to surface ANGLE
NspanSpanwise panels between this section and the next (0 = use surface default)
SspaceSpanwise spacing for this interval (0 = use surface default)

Section keywords (placed after the SECTION data line)

AFIL  or  AFILE
Followed by a filename on the next line. Loads an airfoil coordinate file (Selig or Lednicer format) and extracts camber and thickness distributions. The camber slope is precomputed at uniform chordwise stations for use in the boundary condition.
NACA
Followed by a 4-digit or 5-digit NACA designation (e.g., 4412). Generates camber analytically from the NACA formula. Only 4-digit series is fully supported.
CONTROL
Defines a control surface at this section. Format: 
CONTROL
!Cname   Cgain  Xhinge  HingeVec        SgnDup
flap     1.0    0.75    0.0 0.0 0.0     1.0
  • Cname — control surface name (matched across sections)
  • Cgain — gain factor (sign reversal for differential deflection)
  • Xhinge — hinge location as x/c (positive = trailing edge, negative = leading edge)
  • HingeVec — hinge axis direction (0 0 0 = use default streamwise axis)
  • SgnDup — sign of deflection for the YDUPLICATE'd copy (+1 or -1)
CLAF
Followed by one value: the section lift-curve-slope factor. Default is 1.0, meaning the theoretical 2*pi*alpha is used. Reduce for thick airfoils or set to match experimental CL_alpha/CL_alpha_theory.
CDCL
Defines a piecewise-parabolic viscous drag polar for this section. Format: 
CDCL
!CL1   CD1   CL2   CD2   CL3   CD3
-0.5  0.020  0.1  0.012  1.2  0.025
Provides three (CL, CD) breakpoints that define two parabolic arcs for viscous drag estimation.
DESIGN
Defines a design variable sensitivity at this section. Format: 
DESIGN
!Dname   Dgain
twist    1.0
Design variables perturb the camber line for optimization studies.

4.4 BODY definition

Bodies are modeled using source and doublet line singularities along the body axis. 

BODY
Fuselage                     ! Body name
20   1.0                     ! Nbody  Bspace
BFIL
Followed by a filename. Loads a body profile file containing rows of X R (axial position, radius). The body is assumed axisymmetric.

Body definitions also accept YDUPLICATE, TRANSLATE, and SCALE keywords with the same semantics as surfaces.

Note: Body elements are implemented but not yet fully validated. Cases with significant body influence (e.g., bd, bd2, bdc) may show 10-15% discrepancies compared to Fortran AVL.

5. Core API Reference

5.1 Data Types

TypeDescription
AVLConfigComplete configuration parsed from an .avl file. Contains reference quantities, surface definitions, body definitions, and global parameters.
VortexLatticeDiscretized vortex lattice geometry built from AVLConfig. Contains all horseshoe vortex elements, strips, and body nodes.
AVLSolutionMutable struct holding all solution results: force coefficients, stability derivatives, circulation distribution, per-strip loads, and operating-point parameters.
RunCaseOperating-point specification defining the variable-to-constraint mapping and parameter values (alpha, beta, Mach, CG, etc.).
AICDataCached aerodynamic influence coefficient matrix with its LU factorization. Reusable across multiple run cases at the same Mach number.
VortexElementImmutable struct for a single horseshoe vortex panel: bound leg endpoints, control point, normal vectors, camber slopes, and parent indices.
StripMutable struct for a spanwise station: leading-edge geometry, chord, width, incidence, viscous polar, and per-strip result fields.

5.2 Functions

High-level drivers

FunctionDescription
solve_case(avl_file; alpha, beta, mach, cl_target, verbose) All-in-one solver. Reads an .avl file and returns an AVLSolution. Set alpha for a fixed angle of attack, or cl_target to trim alpha for a target CL.
solve_case(config, rc; verbose) Lower-level solver taking a pre-built AVLConfig and RunCase.
solve_multiple(avl_file, run_file; verbose) Solve all run cases defined in a .run file. Returns Vector{AVLSolution}.
get_lattice(avl_file) Read config and build lattice without solving. Returns (AVLConfig, VortexLattice). Useful for geometry inspection.
get_nodal_forces(sol, vl, config; qinf) Extract dimensional nodal forces for FEM coupling. Returns a vector of (x, y, z, Fx, Fy, Fz) tuples.

Pipeline functions

FunctionDescription
read_avl(filepath) Parse an .avl geometry file. Returns AVLConfig.
build_lattice(config) Build the vortex lattice from surface/section definitions. Returns VortexLattice.
setup_body!(vl, config) Build body source/doublet model. Returns (src_u, dbl_u, wcsrd_u, wvsrd_u) matrices for body-wing interaction.
setup_aic(vl, config) Assemble and LU-factorize the AIC matrix. Returns AICData. This is the O(N^3) step.
solve_unit_rhs(vl, config, aic; wcsrd_u) Solve for unit-freestream circulations. Returns (gam_u0, gam_u_d) where gam_u0[i,iu] is circulation for unit component iu (u,v,w,p,q,r), and gam_u_d holds control-deflection sensitivities.
exec_case!(sol, vl, config, aic, gam_u0, gam_u_d, rc; src_u, wvsrd_u) Run the Newton trim iteration. Updates sol in-place with all forces, moments, and derivatives.

Post-processing functions

FunctionDescription
trefftz_drag!(sol, vl, config) Compute Trefftz-plane induced drag, far-field CL, and span efficiency. Called automatically by exec_case!.
compute_forces!(sol, vl, config, aic, gam_u0, gam_u_d) Compute near-field forces via Kutta-Joukowski integration. Called automatically by exec_case!.
body_forces!(sol, vl, config, src, src_u) Compute body axial forces. Called automatically when bodies are present.

Output functions

FunctionDescription
print_total_forces(io, sol, vl, config) Print total force and moment summary (equivalent to AVL's FT command).
print_strip_forces(io, sol, vl, config) Print per-strip force distribution (equivalent to AVL's FS command).
print_stability_derivatives(io, sol, vl, config) Print stability derivative matrix (equivalent to AVL's ST command).
write_forces_file(filename, sol, vl, config) Write total forces to a text file.
write_strip_file(filename, sol, vl, config) Write strip loading distribution to a text file.
write_element_file(filename, sol, vl) Write per-element data (positions, DCP) to a text file.

6. Constraint System

The constraint system maps operating variables to constraint targets. This is how JAVL.jl handles trim problems: you specify which variable is free and what condition it must satisfy.

6.1 Variables and Constraints

The RunCase struct contains an icon vector that maps each variable to its constraint. The mapping is: rc.icon[iv] = ic, meaning "variable iv is constrained by condition ic".

Variable indices (what can vary)

ConstantIndexVariable
IVALFA1Angle of attack (alpha)
IVBETA2Sideslip angle (beta)
IVROTX3Roll rate (p)
IVROTY4Pitch rate (q)
IVROTZ5Yaw rate (r)

Control surface deflections occupy indices IVTOT+1 through IVTOT+ncontrol.

Constraint indices (what condition to satisfy)

ConstantIndexConstraint
ICALFA1Set alpha to a specified value
ICBETA2Set beta to a specified value
ICROTX3Set roll rate to a specified value
ICROTY4Set pitch rate to a specified value
ICROTZ5Set yaw rate to a specified value
ICCL6Achieve target CL
ICCY7Achieve target CY
ICMOMX8Achieve target rolling moment (Cl = 0 for trim)
ICMOMY9Achieve target pitching moment (Cm = 0 for trim)
ICMOMZ10Achieve target yawing moment (Cn = 0 for trim)

Control surface constraints occupy indices ICTOT+1 through ICTOT+ncontrol, mapping each control to its deflection value.

6.2 Common Constraint Examples

Fixed alpha (default)

rc.icon[AVL.IVALFA]   = AVL.ICALFA      # alpha is set directly
rc.conval[AVL.ICALFA] = deg2rad(5.0)   # alpha = 5 degrees

Trim alpha to achieve CL = 0.5

rc.icon[AVL.IVALFA] = AVL.ICCL   # alpha is free, driven by CL target
rc.conval[AVL.ICCL] = 0.5        # target CL = 0.5

Trim elevator to achieve Cm = 0 (pitch trim)

# Assuming 'elevator' is control #1
rc.icon[AVL.IVALFA]         = AVL.ICALFA
rc.conval[AVL.ICALFA]       = deg2rad(5.0)    # fixed alpha
rc.icon[AVL.IVTOT + 1]      = AVL.ICMOMY     # elevator free, constrained by Cm
rc.conval[AVL.ICMOMY]       = 0.0             # target Cm = 0

Trim alpha and elevator simultaneously

rc.icon[AVL.IVALFA]         = AVL.ICCL       # alpha -> CL = 0.5
rc.conval[AVL.ICCL]         = 0.5
rc.icon[AVL.IVTOT + 1]      = AVL.ICMOMY     # elevator -> Cm = 0
rc.conval[AVL.ICMOMY]       = 0.0
Note: The Newton iteration solves the coupled system simultaneously. Convergence typically requires 3-5 iterations. The default tolerance is 2e-5 with a maximum of 20 iterations.

7. Module Architecture

JAVL.jl is organized as a single Julia module (AVL) composed of source files that mirror the structure of the original Fortran codebase. All files are included from AVL.jl, the top-level module file.

src/ AVL.jl -- Main module file; includes all sub-files and exports types.jl -- Data structures: AVLConfig, VortexLattice, AVLSolution, RunCase, etc. math_utils.jl -- Vector math helpers: cross3, normalize3, dot3, etc. splines.jl -- Cubic spline and Akima interpolation (matching AVL's spline.f) airfoil.jl -- Airfoil file parsing, camber/thickness extraction, NORMIT normalization spacing.jl -- Panel spacing functions: SPACER (spanwise), CSPACER (chordwise) input.jl -- .avl file parser (matching ainput.f) mass.jl -- Mass file parser (.mass) geometry.jl -- Vortex lattice construction from surface/section definitions body.jl -- Body source/doublet line model (SRDSET, VSRD, BDFORC) aic.jl -- AIC matrix assembly, horseshoe vortex velocity kernel solver.jl -- AIC factorization, unit-solution computation aero.jl -- Near-field force and moment calculations (Kutta-Joukowski) trefftz.jl -- Trefftz-plane induced drag integration trim.jl -- Newton trim iteration, constraint system output.jl -- Output formatting: print_total_forces, print_strip_forces, etc. driver.jl -- High-level driver functions: solve_case, solve_multiple, get_nodal_forces

7.1 Key Design Decisions

  • Julia structs replace Fortran COMMON blocks. All state is encapsulated in typed structures, making the data flow explicit.
  • NTuple{3,Float64} is used for 3-vectors (positions, directions, velocities). Being immutable, they are stack-allocated and avoid heap pressure.
  • Mutable structs are used for AVLSolution, Strip, and AVLConfig where in-place updates are needed.
  • AICData caches the LU factorization so multiple run cases can share the expensive O(N^3) matrix factorization.
  • No external dependencies — only LinearAlgebra and Printf from the standard library.

7.2 Computation Pipeline

The solver follows this pipeline, matching the original Fortran flow:

  1. Parse — read_avl reads the .avl file into AVLConfig
  2. Build lattice — build_lattice creates vortex elements and strips via SPACER/CSPACER spacing
  3. Build bodies — setup_body! creates source/doublet nodes (if bodies present)
  4. AIC assembly — setup_aic assembles the influence matrix from horseshoe vortex velocities and LU-factorizes it
  5. Unit solutions — solve_unit_rhs back-substitutes for each unit freestream component
  6. Trim iteration — exec_case! runs Newton iteration: superpose unit solutions, compute forces, check constraints, update variables
  7. Post-process — compute_forces! and trefftz_drag! fill in all force/moment/derivative fields

8. Running the Validation Suite

JAVL.jl includes a comprehensive validation suite that compares its output against reference data from Fortran AVL 3.52.

8.1 Running the tests

julia test/test_all_cases.jl

8.2 What it does

  1. Reads all .avl files from the validation/cases/ directory (55 cases)
  2. Solves each case at alpha = 5 degrees using solve_case
  3. Compares CL, CDff, and span efficiency against test/avl_reference.csv
  4. Prints a comparison table with absolute and percentage errors
  5. Reports summary statistics: mean error, max error, number within 1%/5% thresholds

9. Differences from Fortran AVL

9.1 What is the same

  • Horseshoe vortex kernel with Leishman R^4 core model
  • Cubic Akima spline interpolation for airfoil camber
  • SPACER/CSPACER panel spacing algorithms
  • Spanwise fudging for multi-section surfaces
  • Trefftz-plane drag integration
  • Kutta-Joukowski near-field force computation
  • Newton iteration for trim with the same constraint system
  • Prandtl-Glauert compressibility correction
  • Same .avl input file format

9.2 What is different

AspectFortran AVLJAVL.jl
Language Fortran 77/90 Pure Julia — no compilation step, no Fortran dependencies
Data architecture COMMON blocks with global arrays Typed structs with explicit ownership
3-vectors Separate X, Y, Z arrays Immutable NTuple{3,Float64} (stack-allocated)
Linear algebra Hand-coded LU decomposition Julia's LinearAlgebra.lu() (LAPACK)
Interactive UI Full terminal-based menu system Programmatic API only (no interactive menus)
Plotting Built-in PostScript/X11 geometry viewer Not included (use Julia plotting packages)
BODY model Fully validated Implemented but with remaining discrepancies (10-15% on body cases)

9.3 Fortran flags not yet toggleable

Two Fortran compile-time flags default to FALSE and are not yet exposed as user-configurable options in JAVL.jl:

  • LTRFORCE — When TRUE, includes trailing-leg on-surface forces in the near-field force computation. Currently always FALSE in JAVL.jl (matching the AVL default).
  • LNFLD_WV — When TRUE, includes body-source induced velocity in the near-field force computation. Currently always FALSE in JAVL.jl.
